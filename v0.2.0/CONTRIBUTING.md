# Контрибьют

Спасибо что хочешь помочь. Несколько правил чтобы PR прошёл быстро.

## С чего начать

1. **Форкни** репо на GitHub
2. **Клонируй** свой fork локально
3. Заведи ветку: `git checkout -b fix/<краткое-имя>` или `feat/<краткое-имя>`
4. Сделай правки
5. Прогони форматтер: `nix fmt`
6. Прогони `nix flake check` — должно пройти без ошибок
7. Запушь, открой PR на `Nhrust/nixos-config:main`

## Что приветствуется

| Тип PR | Куда смотреть |
|---|---|
| Bug-fix существующего модуля | `modules/system/` или `modules/user/` |
| Поддержка нового железа | `modules/system/hardware/` (новый файл) |
| Новый профиль системы | `modules/system/profiles/` |
| Готовый «комплект» (extra) | `extras/<имя>.nix` + обновить `extras/README.md` |
| Улучшение документации | `docs/` или `README.md` |
| Bug-fix в dotfiles | `modules/user/dotfiles/...` |
| Новый алиас в fish | `modules/user/shell/fish.nix` |

## Что НЕ приветствуется

- Изменения в **`hosts/_template/settings.nix`** — добавляй только новые
  опциональные поля с разумными дефолтами. Не меняй существующие.
- **Личные предпочтения** (любимая тема, бинды) — заведи свой fork, не PR в upstream.
- **`flake.lock` updates** — это делает мейнтейнер раз в неделю.
- **Гигантские PR** «полная переделка системы». Лучше серия маленьких.

## Стиль кода

### Nix

- 2 пробела отступы (см. `.editorconfig`)
- `nix fmt` обязательно — это `nixfmt-rfc-style`
- Комментарии на русском или английском (репо двуязычный)
- Каждый модуль с заголовком-комментарием: что делает, какие settings.* читает
- Условные подключения через `lib.mkIf` или `lib.optionalAttrs`, не через `if then else`
  где это возможно

### Документация

- Markdown
- Заголовки последовательные (`#` → `##` → `###`)
- Таблицы для матриц и сравнений
- Кодблоки с указанием языка (`fish`, `nix`, `bash`)

### Коммит-сообщения

```
<тип>: <краткое описание>

<подробности если нужны>
```

Типы:
- `fix:` — багфикс
- `feat:` — новая фича
- `refactor:` — рефакторинг без изменения поведения
- `docs:` — только docs
- `release:` — релиз (только для мейнтейнера)

Примеры:
- `fix: hypridle stops on suspend-then-hibernate`
- `feat(extras): add media.nix with Spotify/VLC/OBS`
- `docs: clarify kbLayouts default in CUSTOMIZATION.md`

## Перед PR проверь

```fish
nix develop                   # войти в dev shell
nix fmt                       # форматнуть всё
deadnix                       # найти неиспользуемый код
statix check                  # antipatterns
nix flake check               # проверить флейк
```

## Тестирование

Если меняешь системный модуль — желательно собрать на твоей машине:

```fish
sudo nixos-rebuild test --flake "path:.#$(hostname)"
```

Если ломает — откатишься на ребуте, реальный switch не сделан.

Если меняешь home-manager модуль — то же самое:

```fish
home-manager build --flake "path:.#$(hostname)"
```

## Что мейнтейнер сделает с твоим PR

1. Прочитает diff
2. Проверит `nix flake check` (вручную или через CI когда появится)
3. Может попросить исправлений в комментариях
4. После merge — добавит запись в `CHANGELOG.md` под следующим релизом
5. Закроет PR с пометкой релиза

## Вопросы

Открывай issue с тегом `question` — отвечу когда увижу.
Для критических багов — тег `urgent`.
